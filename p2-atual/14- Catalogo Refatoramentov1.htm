Objetivos da seção

•    Conhecer os principais refatoramentos realizados em software e acompanhar vários pequenos exemplos de refatoramentos de software, que podem ser usados para transformar um software de qualidade pobre em um de melhor qualidade

Refatoramento

•    Como programador, você precisa treinar seu nariz para detectar “mau cheiro” em código

•    Quanto maior o sistema (ou mais complexo) pior o efeito de um código mal feito

•    Refatoramento altera um programa mas sem afetar a funcionalidade que ele oferece

•    Refatoramento sempre deve ser feito apoiando-se em Testes de Unidade para assegurar-se de que as transformações no código não quebrem código que funciona

•    Refatoramento: Extract Method

›       Problema: Método longo

i)    Sempre que sentir a necessidade de escrever um comentário no código, é sinal de que um método deve ser criado

ii)Métodos longos dificultam o reuso e o seu entendimento

iii)         Você deve preferir ter métodos curtos e com nomes significativos

›       Solução

i)    99% das vezes que você precisar reduzir o tamanho de um método será usado o refatoramento Extract Method

›       Exemplo: Extract Method (pag 112 a 115)

void imprimeDebito() {

   imprimeBanner();

 

   //imprime detalhes

   System.out.println ("nome: " + _nome);

   System.out.println ("quantia: " + getValorDivida());

}

void imprimeDebito() {

   imprimeBanner();

   imprimeDetalhes(getValorDivida());

}

 

void imprimeDetalhes (double valorDoDebito) {

   System.out.println ("nome: " + _nome);

   System.out.println ("quantia: " + valorDoDebito);

}

 

Refatoramento: Move Method e Move Field

›       Problema: Método usa mais dados de outras classes do que dele mesmo para preencher sua responsabilidade

i)    Responsabilidades atribuídas incorretamente

ii)Não aplicar o padrão Information Expert aumenta o acoplamento entre as classes e diminui a coesão das classes

iii)         Você deve definir corretamente os campos e métodos de suas classes

›       Solução

i)    Move Method e Move Field são refatoramentos muito usados, mas nem sempre é fácil decidir de onde ou para onde mover métodos e campos das classes

ii)Dica: começar do mais simples para o mais nebuloso

Refatoramento: Replace Temp with Query

›       Problema: Muitas variáveis temporárias

i)    Variáveis temporárias podem contribuir para ter métodos longos

ii)É bom evitar a complexidade de gerenciar variáveis temporárias

›       Solução

i)    Variáveis temporárias normalmente são usadas para guardar o resultado de uma expressão, nesse caso, pode-se extrair esta expressão para um método e substituir a variável pela chamada do método

Refatoramento: Extract Superclass e Extract Interface

›       Problema: Baixo reuso

i)    Duas classes podem ter implementações/interfaces similares

ii)Contribui para ter duplicação de código e classes grandes

›       Solução 1: Extract Superclass

i)    Responsabilidades comuns (código) entre duas ou mais classes podem ser extraídas para uma nova classe (superclasse), fazendo estas classes se relacionarem por meio de herança

ii)Favorece uso de polimorfismo

›       Solução 2: Extract Interface

i)    Agrupar responsabilidades em Interfaces

ii)Isso é bom para o caso de uma classe que pode assumir responsabilidades distintas em diferentes situações

iii)         Além disso, quando mais de uma classe pode assumir o mesmo subconjunto de responsabilidades sem estarem semanticamente relacionadas

iv)         Favorece uso de polimorfismo com menor acoplamento entre as classes

›       Exemplo pag 342 e 343

class FolhaDePagamento {

   ...

 

   double encargos(Empregado emp, int dias){

      int base = emp.getTaxa() * dias;

      if(emp.hasHabilidadeEspecial())

         return base * 1.05;

      else

         return base;

   }

}

 

class Empregado {

   ...

 

   int getTaxa(){...};

   boolean hasHabilidadeEspecial(){...};

   //outros métodos (responsabilidades)...

}

 

interface Faturavel{

   public int getTaxa();

   public boolean hasHabilidadeEspecial();

}

 

class Empregado implements Faturavel {

   ...

}

 

class FolhaDePagamento {

   ...

 

   double encargos(Faturavel emp, int dias){

      int base = emp.getTaxa() * dias;

      if(emp.hasHabilidadeEspecial())

         return base * 1.05;

      else

         return base;

   }

}

 

Palavras Finais

•    Uma faceta importante do refactoring é o ritmo

›       Testa, mude um pouco, testa, mude um pouco, ...

›       Este ritmo permite fazer refatoramento de forma simples, rápida e segura
