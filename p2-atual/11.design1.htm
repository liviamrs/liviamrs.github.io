O que é design e como fazê-lo?

    É uma das partes mais difíceis da programação

    Consiste em criar abstrações

       Isto significa quatro coisas:

i)    Quais classes devem ser criadas?

ii)Quais responsabilidades (métodos) devem ser assumidas por cada classe?

iii)        Quais são os dados (atributos) que uma classe deve encapsular?

iv)        Quais são os relacionamentos entre tais classes e objetos dessas classes?

    Algumas orientações gerais incluem:

       Partimos da especificação do sistema (fase de análise do software)

i)    Especificação de alto nível, onde são identificados os elementos relevantes do mundo real a serem modelados no mundo de software

       Mapear elementos do mundo real para elementos da linguagem:

i)    Substantivos são objetos/atributos

ii)Verbos são métodos

iii)        Classes são fábricas de objetos

iv)        Classes se relacionam entre si por meio de herança, composição e/ou interfaces

    Criar boas abstrações é difícil e vem com experiência

    Porém, algumas regras básicas ajudarão a adquirir a experiência mais rapidamente

       Queremos usar os padrões GRASP que ajudam na atribuição de responsabilidades no sistema

Falando em responsabilidades

    Distribuir responsabilidades entre objetos (ou classes) é uma tarefa crucial para um bom projeto OO

    Responsabilidades são obrigações de um tipo ou de uma classe

i)    Obrigações de fazer algo

(1)                 Fazer algo a si mesmo

(2)                 Iniciar ações em outros objetos

(3)                 Controlar ou coordenar atividades em outros objetos

ii)Obrigações de conhecer algo

(1)                 Conhecer dados encapsulados

(2)                 Conhecer objetos relacionados

(3)                 Conhecer coisas que ele pode calcular

    Por exemplo:

       Uma conta bancária tem a responsabilidade de sacar (fazer algo)

       Uma conta bancária tem a responsabilidade de saber seu saldo (conhecer algo)

    Atenção: responsabilidades não remete unicamente para um método, mas métodos são usados para implementar responsabilidades

Padrões GRASP

    Por que não usar as boas soluções em outros projetos?

       Padrões de software ou padrões de projeto

    Mas o que é um padrão de projeto?

... cada padrão descreve um problema que ocorre frequentemente e então descreve o cerne da solução ao problema de forma a poder reusar a solução milhões de vezes em situações diferentes.

... descrevem soluções para problemas recorrentes no desenvolvimento de sistemas de software orientados a objetos. Um padrão de projeto estabelece um nome e define o problema, a solução, quando aplicar esta solução e suas consequências.

    GRASP = General Responsibility Assingment Software Pattern

     Incluem 9 padrões

       Queremos explorar 4 padrões básicos: Expert, Creator, High Coesion, Low Coupling

    Veja mais:

       Material prof. Jacques Sauvé

       Material prof. Nazareno Andrade

Regra número 0: Simplicidade

    Também conhecida como KISS (Keep It Simple, Stupid)

    Lembre de Saint-Exupéry:

       Atingimos a perfeição não quando nada pode acrescentar-se a um projeto mas quando nada pode retirar-se

       Esta é a MegaRegra

Regra 1: Especialista da informação (Expert)

    [problema] Qual é o princípio mais fundamental para atribuir responsabilidades?

       [solução] É este: Atribuir uma responsabilidade ao expert de informação - a classe que possui a informação necessária para preencher a responsabilidade

       Exemplo: Entre as seguintes classes do mundo bancário, (Agencia, Conta, ContaCaixa, ContaSimples, Extrato, ExtratoHTML, Moeda, Movimento, Real, Transacao), quem deve ser responsável pela responsabilidade Localizar a conta com certo número?

i)    Perguntamos: onde estão guardadas as Contas? (onde estão os dados)

(1)                 Estão na Agencia

(2)                 Portanto, a classe Agencia deve ter a responsabilidade (através do método localizarConta(int número))

       [Consequências]

i)    O encapsulamento e a ocultação da informação são mantidos, já que objetos usam sua própria informação para cumprir suas responsabilidades

ii)Leva a fraco acoplamento entre objetos e sistemas mais robustos e fáceis de manter

iii)        Leva a alta coesão, já que os objetos fazem tudo que é relacionado à sua própria informação

    Também conhecido como:

       "Quem sabe, faz"

       "Expert"

       "Animação" (objetos são vivos e podem assumir qualquer responsabilidade, mesmo que sejam passivos no mundo real)

i)    Exemplo: No mundo bancário real, uma agência é algo passivo e não "localiza contas"

       "Eu mesmo faço"

       "Colocar os serviços junto aos atributos que eles manipulam"

Regra 2: Criador (Creator)

    [problema] Quem deve criar novas instâncias de uma classe?

[solução] Atribua à classe B a responsabilidade de criar instâncias de uma classe A se:

       B agrega objetos da classe A

       B contém objetos da classe A

       B registra instâncias da classe A

       B usa muito objetos da classe A

       B possui os dados usados para inicializar A

    B é um criador (creator) de objetos da classe A

    Se mais de uma opção se aplica, escolha o B que agregue ou contenha objetos da classe A

    Exemplo: mundo bancário

       De quem seria a responsabilidade de criar objetos do tipo Pessoa, sabendo que cada Conta tem um cliente que será representado por Pessoa? Da classe Conta!

    [consequência] O principal benefício desse padrão é deixar o objeto criado próximo ao seu criador. Isso diminui o acoplamento. Ou seja, quem cria o objeto é quem vai precisar usá-lo.

Regra 3: Alto Coesão (High Coesion)

    A coesão mede quão relacionados ou focados estão as responsabilidades da classe

       Como gerenciar a complexidade?

i)    Também chamado coesão funcional

    O [problema]:

       Uma classe com baixa coesão faz muitas coisas não relacionadas e leva aos seguintes problemas:

i)    Difícil de entender

ii)Difícil de reusar

iii)        Difícil de manter

iv)        É delicada: constantemente sendo afetada por outras mudanças

       Uma classe com baixa coesão assume responsabilidades que pertencem a outras classes e deveriam ser delegadas a outras

    A [solução]:

       Atribuir responsabilidades que mantenham alta coesão

       Uma classe deve ter um pequeno número de variáveis e métodos que manipulam essas variáveis

i)    Coesão máxima ocorre quando todas as variáveis são manipuladas por todos os métodos da classe

ii)Quando a coesão é alta dizemos que os métodos e variáveis da classe são co-dependentes e formam uma unidade lógica

    Olhar coesão pode ajudar a quebrar classes

       Quando descobrimos que a classe tem responsabilidades demais

       Quando descobrimos que elas manipulam conjuntos disjuntos de variáveis

    Exemplo: O que acha da classe que segue?

class Angu {

    List números;

    String texto;

 

    public static int acharPadrão(String padrão) {

        // ... lida com texto aqui

    }

    public static int média() {

        // ... lida com números aqui

    }

    public static outputStream abreArquivo(string nomeArquivo) {

        // ...

    }

}

 

class Xpto extends Angu { // quer aproveitar código de Angu

    ...

}

 

    [consequência]

  Classes com alta coesão são preferíveis: mais robustas, mais confiáveis mais fáceis de serem reusadas e entendidas;

  Alta coesão prove o design modular;

       A coesão baixa resulta em código difícil de manter, de testar, de reusar e de entender!

Princípio da responsabilidade única

       Classes devem ser pequenas. Quanto é pequeno?

       Mede-se o tamanho de uma classe pela quantidade de responsabilidades que elas têm

i)    Deve ser possível descrever a classe rapidamente (com umas 25 palavras)

ii)Dar um nome à classe deve ser fácil   

(1)                 O nome descreve suas responsabilidades

(2)                 O nome ajuda a determinar o tamanho de classe

(3)                 Se estiver difícil nomear a classe, desconfie de que existe um problema!

iii)        O ideal é que uma classe deve ter apenas uma razão para mudar

(1)                 Classe que compila e imprime relatórios

(2)                 Pode mudar por diferentes razões

(a)     Esta classe deve ser quebrada em pelo menos 2 outras!

 

Regra 4: Baixo Acoplamento (Low Coupling)

    O acoplamento é uma medida de quão fortemente uma classe está conectada, possui conhecimento ou depende de outra classe

       Como minimizar dependências e, consequentemente, maximizar o reuso?

    [O problema]:

       Uma classe que possui forte acoplamento com outras causa as seguintes dificuldades de manutenção:

i)    Mudanças a uma classe relacionada forçam mudanças locais à classe

ii)A classe é mais difícil de entender isoladamente

iii)        A classe é mais difícil de ser reusada, já que depende da presença de outras classes

    [A solução]: Atribuir responsabilidades de forma a minimizar o acoplamento

       Minimizar acoplamento é um dos princípios de ouro do projeto OO

       Acoplamento de manifesta de várias formas:

i)    X tem um atributo que referencia uma instância de Y

ii)X tem um método que referencia uma instância de Y

(1)                 Pode ser parâmetro, variável local, objeto retornado pelo método

iii)        X é uma subclasse direta ou indireta de Y

iv)        X implementa a interface Y

       A herança é um tipo de acoplamento particularmente forte

       Não se deve minimizar acoplamento criando algumas poucas classes monstruosas (God classes)

iii)  Quanto maior a coesão, menor o acoplamento e vice-versa 

 [Consequências]:

       Menor dependência entre as classes

       Mudanças em uma classe tem pouco impacto em outras classes

       Maio reuso

    Exemplo: Ordenação de registros de alunos por matrícula
